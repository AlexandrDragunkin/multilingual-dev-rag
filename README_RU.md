# multilingual-dev-rag

Локальный многоязычный RAG по исходному коду и технической документации. Гибридный поиск в процессе — вектор плюс полнотекстовый, — с FTS для кириллицы и других не-ASCII письменностей.

## Зачем это

Обычный локальный RAG берёт многоязычную модель эмбеддингов, добавляет полнотекстовый индекс и называет это гибридным поиском. Работает — пока корпус не на английском.

Встроенный движок ([zvec](https://github.com/alibaba/zvec)) поставляется с токенайзером `standard`, который **выбрасывает не-ASCII токены целиком** (на Windows и Linux — на macOS, что любопытно, не выбрасывает), и фильтром `lowercase`, который **складывает регистр только для ASCII**. Ошибки при этом нет. Запрос `воркер` возвращает ноль строк, запрос `RabbitMQ` возвращает строки, и гибрид тихо вырождается в чистый вектор на половине корпуса. В логах — ни слова.

Платформенный разрыв — половина истории похуже. Пакет, полагающийся только на `standard`, *выглядел* бы рабочим на маке разработчика и молча ломался при выкатке на Linux-сервер — тот же код, тот же движок, но разное поведение токенайзера по ОС. Нижеследующий фикс не зависит от того, на какой стороне этого разрыва вы оказались: он работает везде, а там, где `standard` уже умеет кириллицу, он просто избыточен.

## Как устроен фикс

Сменой токенайзера это не лечится. В движке их ровно три, и каждый ломается по-своему:

| Токенайзер | Кириллица | Идентификаторы | Пунктуация |
| --- | --- | --- | --- |
| `standard` | **выбрасывает**¹ | режет верно: `obj_k3_gab3` → `obj`, `k3`, `gab3` | отделяет верно |
| `whitespace` | сохраняет | **не режет** | **приклеивает**: `RabbitMQ.` — один токен |
| `jieba` | **рвёт на фрагменты** | режет | отделяет |

¹ На Windows и Linux. На macOS `standard` кириллицу **находит** — единственная из трёх платформ, где он это делает (проверено CI-матрицей на zvec 0.5.1). Поэтому обход через `text_fts` ниже на macOS избыточен, но безвреден: его дороги не задваивают совпадения, потому что латиница в `text_fts` не попадает.

`jieba` выглядит спасением ровно до первой проверки на объёме: на пяти документах он давал 7 верных ответов из 9, а на реальных 837 чанках выродился в шум — запрос словом из заголовка одного плана возвращал три чужих, по одному на каждый регистр. Хуже `standard`: тот хотя бы честно молчал.

Отсюда решение: **не выбирать лучший токенайзер, а взять два** — каждый на то, что он умеет. В индексе два поля, и запрос идёт только туда, где на него могут ответить:

| Поле | Содержимое | Токенайзер | Отвечает за |
| --- | --- | --- | --- |
| `text` | оригинал, дословно | `standard` | ASCII-слова и идентификаторы — `obj_k3_gab3` находится по `obj`, `k3`, `gab3` |
| `text_fts` | только не-ASCII токены, casefold, без пунктуации | `whitespace` | кириллица, греческий, латиница с диакритикой, арабский — в любом регистре |

`text` остаётся побайтово исходным: именно его поиск возвращает вам. `text_fts` — производное и служебное.

Латиница из `text_fts` исключена **намеренно**. Если бы оба поля матчили `RabbitMQ`, документ с ним получал бы двойной вес в Reciprocal Rank Fusion и обходил документы, отвечающие на русскую половину запроса. Непересекающиеся дороги делают слияние честным.

Складывание регистра происходит в Python, а не в движке: `casefold()` справляется с `ß`/`SS` и греческой финальной сигмой, чего ASCII-фильтр `lowercase` не умеет.

## Что внутри

- **Гибридный поиск** — вектор плюс полнотекстовый, слияние через Reciprocal Rank Fusion.
- **FTS, переживающий не-ASCII** — проверено сквозным зондом на русском, немецком, греческом, французском, арабском.
- **Без серверов** — zvec работает в вашем процессе. Ни Qdrant, ни Ollama, ни Docker.
- **Профили** — корпус описывается в JSON, пакет остаётся универсальным.
- **MCP-сервер** — отдаёт инструмент `rag_search` в Claude Code и другие MCP-клиенты.

## Требования

**Python 3.10–3.14, 64-битный.** Всё остальное следует из вилок zvec — **sdist у него нет**, поэтому если под вашу платформу нет готовой сборки, собрать из исходников не выйдет.

| Платформа | Вилка | Статус |
| --- | --- | --- |
| Windows x86_64 | `win_amd64` | **проверено** — здесь и разрабатывалось |
| Linux x86_64 / aarch64 | `manylinux_2_28` | **проверено** — CI зелёный (Python 3.10–3.13) |
| macOS Apple Silicon (11+) | `macosx_11_0_arm64` | **проверено** — CI зелёный (Python 3.10–3.13) |
| macOS Intel | — | **не установится** |
| Любой 32-битный Python | — | **не установится** |

Все три платформы с вилкой прогоняют полный набор тестов в CI, включая семь интеграционных тестов, бьющих в живой движок zvec.

На платформе без вилки `pip install` выдаёт `from versions: none`. Это означает *не та платформа*, а не *пакета нет* — никакие сборочные инструменты не помогут.

Прочее:

- **457 МБ** на диске под модель эмбеддингов, скачивается при первой индексации в `~/.cache/huggingface/`. Общая на все проекты, к индексу отношения не имеет.
- На macOS индекс ляжет в `~/.local/share/dev-rag/` (путь XDG), а не в родной `~/Library/Application Support`. Нужен родной — задайте `ZVEC_DB_PATH`.

## Установка

```bash
pip install multilingual-dev-rag
```

Модель по умолчанию (`paraphrase-multilingual-MiniLM-L12-v2`, Apache-2.0) скачивается с Hugging Face при первом использовании и в пакет не вшита.

## Настройка

Две переменные окружения:

```bash
# Обязательная: репозиторий, который индексируем.
export DEV_RAG_ROOT=/путь/к/вашему/репозиторию

# Необязательная: какие файлы считать. Встроенные: generic (по умолчанию), k3_mebel.
# Либо путь к своему JSON-профилю.
export DEV_RAG_PROFILE=generic
```

`DEV_RAG_ROOT` **не угадывается** по расположению пакета, и это сделано намеренно. В ранней версии дефолт считался от каталога установки; когда догадка не совпадала, globs не находили ничего, индексация печатала `0 chunks` и завершалась успешно.

Эту догадку заменили две проверки. Отсутствующий корень или не-каталог приводит к падению до начала работы. Корень, который существует, но не содержит ни одного файла под паттерны профиля, заставляет `dev-rag-index` **выйти с кодом 2**, напечатав оба значения: пустой индекс, собранный «успешно», выглядит здоровым ровно до первого поиска — который ничего не найдёт и тоже промолчит.

### Где лежит индекс

На пользователя и на корпус — но не внутри пакета:

```text
Linux/macOS   ${XDG_DATA_HOME:-~/.local/share}/dev-rag/<репозиторий>-<хэш>/zvec_data
Windows       %LOCALAPPDATA%\dev-rag\<репозиторий>-<хэш>\zvec_data
```

`<хэш>` считается от абсолютного пути корня, поэтому **одна установка обслуживает несколько репозиториев**, не путая их. Направите `DEV_RAG_ROOT` в другое место — получите отдельный индекс, а не затёртый: иначе поиск начал бы уверенно отвечать не из того корпуса.

`ZVEC_DB_PATH` перекрывает расположение целиком.

Индекс — производные данные: в нём чанки того, что вы проиндексировали, он пересобирается командой `dev-rag-index` и в системе контроля версий ему не место.

## Использование

```bash
# Проиндексировать корпус целиком
dev-rag-index --category all --force

# Одну коллекцию
dev-rag-index --category docs --force

# Поиск из консоли
dev-rag-search "ферма воркеров" --collection docs
```

> **Первый запуск дольше остальных.** Модель эмбеддингов в пакет не вшита:
> `sentence-transformers` скачивает её с Hugging Face при первом обращении —
> 457 МБ в `~/.cache/huggingface/hub/` (на Windows —
> `C:\Users\<вы>\.cache\huggingface\hub\`). Это **не** тот каталог, где лежит
> индекс: модель одна на все проекты, индекс — на каждый корпус свой.
> Дальнейшие запуски модель берут из кэша.
>
> Для офлайн-окружения кэш можно перенести с машины, где он уже прогрет, либо
> задать `HF_HOME`.

```python
from dev_rag import search

for r in search('усталость воркера', collection='docs', n=5):
    print(f"{r['score']:.4f}  {r['path']}")
```

### Как MCP-сервер

```json
{
  "mcpServers": {
    "dev-rag": {
      "command": "dev-rag-mcp",
      "env": {
        "DEV_RAG_ROOT": "/абсолютный/путь/к/репозиторию",
        "DEV_RAG_PROFILE": "generic"
      }
    }
  }
}
```

MCP-сервер — дочерний процесс клиента, а не служба: `env` читается один раз при старте. После правки конфига клиент нужно перезапустить.

## Профили

Профиль говорит, какие файлы относятся к какой коллекции. `generic` индексирует markdown и Python где угодно; `k3_mebel` — рабочий пример для CAD-проекта с макросами `.mac`.

```json
{
  "name": "my_project",
  "collections": { "docs": "my_docs", "code": "my_code", "plans": "my_plans" },
  "index_patterns": {
    "docs":  ["docs/**/*.md"],
    "code":  ["src/**/*.py"],
    "plans": ["rfcs/*.md"]
  }
}
```

```bash
export DEV_RAG_PROFILE=/путь/к/my_project.json
```

## Бэкенды

| Бэкенд | Модель | Нужны серверы |
| --- | --- | --- |
| **zvec** (по умолчанию) | `paraphrase-multilingual-MiniLM-L12-v2` (384d) | нет — всё в процессе |
| qdrant (legacy) | `nomic-embed-text` через Ollama (768d) | Qdrant + Ollama |

Ветка с Qdrant появилась раньше zvec и оставлена только как запасная: `nomic-embed-text` англоцентричен и плохо ищет по русскому. Ставится через `pip install multilingual-dev-rag[qdrant]`, включается `RAG_BACKEND=qdrant`.

## Известные ограничения

- **CJK не поддержан.** В китайском и японском нет пробелов между словами, поэтому токенайзер `whitespace` проглотит предложение как один токен. У zvec есть токенайзер `jieba` ровно для этого случая — под него нужно отдельное поле. (На кириллице `jieba` использовать не стоит: он её индексирует, но рвёт на фрагменты, совпадающие со всем подряд. На реальном корпусе ранжировал хуже, чем отсутствие результата вообще.)
- **Турецкие `İ`/`ı` не поддержаны.** Беcрегистровое сравнение там зависит от локали, и ни `lower()`, ни `casefold()` его не решают.
- **Индексация ручная.** Инкрементальная индексация (`--changed-only`) для бэкенда zvec не реализована; после изменений в корпусе нужен полный прогон.
- **Один писатель.** zvec допускает много читателей, но одного писателя: не индексируйте, пока другой процесс держит коллекцию открытой на запись.

## Происхождение

Создан изначально для [K3-Мебель](https://k3-mebel.ru/) — CAD-системы проектирования корпусной мебели — чтобы дать семантический и полнотекстовый поиск по русскоязычной кодовой базе и её документации. Именно там всплыла проблема с FTS по не-ASCII, и поэтому фикс мерялся на настоящем корпусе, а не на игрушечном примере.

Профиль `k3_mebel` — это тот самый исходный корпус, оставленный как рабочий пример ручной настройки профиля: Python-прототипы, макросы `.mac`, справочник API и планы.
