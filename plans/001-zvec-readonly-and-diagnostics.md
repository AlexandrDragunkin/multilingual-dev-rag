# Plan 001 — zvec: read-only открытие коллекции + явные диагностики

**Статус:** предложено (2026-07-17)
**Автор:** Alexandr Dragunkin (по итогам расследования в ARLINE)
**Приоритет:** высокий — блокирует план 002 ARLINE (3-4 ПК, 2+ MCP-клиента)

## Контекст

Выявлено при запуске RAG в проекте ARLINE. Три дефекта в `src/dev_rag/zvec_searcher.py`
и `src/dev_rag/mcp_server.py`, каждый по отдельности даёт правдоподобный, но
неверный ответ. Все три подтверждены воспроизводимыми измерениями, не рассуждениями.

## Предусловия

- zvec-бэкенд (`RAG_BACKEND=zvec`), индекс построен.
- Запрос `dev-rag-search "ферма воркеров" --collection docs` возвращает
  дифференцированные score `0.0159-0.0164` и корректный top-3, **когда
  ни один MCP-клиент не держит коллекцию открытой**.

---

## Дефект 1 (КРИТИЧНО): эксклюзивный read-write lock на коллекцию

**Симптом.** `dev-rag-search` из CLI возвращает «No results found», если запущен
любой MCP-клиент (Zcode, Claude Code, SourceCraft). После закрытия клиента —
работает. Воспроизведено автором 2026-07-17.

**Корень.** Существующая коллекция открывается как
`return zvec.open(coll_path)` (`zvec_searcher.py:59`) — без `CollectionOption`,
то есть в режиме read-write по умолчанию. zvec берёт **эксклюзивный** lock
(`…/zvec_data/<cat>/LOCK`), второй процесс получает
`RuntimeError: Can't lock read-write collection`.

Следствие для ARLINE: при открытом AI-клиенте CLI бесполезен; два MCP-клиента
на одной машине конфликтуют; план 002 (3-4 ПК, по 2+ клиента на машине)
неработоспособен без правки.

### Решение

Поисковый путь открывает существующую коллекцию в режиме read-only:

```python
# zvec_searcher.py:55-59 — заменить
def _open_or_create_collection(category: str) -> Collection:
    coll_path = os.path.join(ZVEC_DB_PATH, category)
    if os.path.exists(coll_path):
        return zvec.open(
            coll_path,
            option=CollectionOption(read_only=True, enable_mmap=True),
        )
    # ... создание (ниже) остаётся read_only=False — это прерогатива индексатора
```

**Важно:** режим создания (`zvec.create_and_open`, `zvec_searcher.py:95-99`)
должен остаться `read_only=False` — это путь индексатора, не поиск. Если
поисковик будет создавать коллекции, открывать их на запись бессмысленно, но
искать через read-only можно. Лучше вообще запретить поисковому пути создавать
коллекцию: если её нет — это ошибка конфигурации, а не повод молча создать
пустую.

### Приёмка

1. Построить индекс: `dev-rag-index --category docs --force` (клиент закрыт).
2. Открыть один MCP-клиент.
3. В отдельном терминале, **не закрывая клиент**, запустить
   `dev-rag-search "ферма воркеров" --collection docs` → должен вернуть те же
   результаты, что и при закрытом клиенте.
4. Открыть второй MCP-клиент → оба возвращают результаты, никто не получает
   «No results found».

Без шага 3-4 правка не считается принятой.

## Дефект 2: тихий отказ в `_get_collection`

**Симптом.** Любая ошибка открытия коллекции (включая lock из дефекта 1)
превращается в «No results found», будто корпус пуст. Пользователь не видит
причины.

**Корень.** `zvec_searcher.py:106-109`:

```python
try:
    _collections[category] = _open_or_create_collection(category)
except Exception:
    return None
```

`return None` → `search()` пропускает категорию (`if coll is None: continue`,
`zvec_searcher.py:133`) → пустой результат. В CLI это печатается как
«No results found» (`cli.py:60-63`).

### Решение

Логировать исключение с уровнем WARNING/ERROR, не глушить. Варианты:

- Минимальный: `except Exception as e: logging.warning('zvec open %s failed: %s', category, e); return None`.
- Лучше: пробросить выше и дать CLI/MCP показать причину. Для MCP вернуть
  осмысленный текст в `_format_results` (например,
  `f"Index unavailable: {e}"`), для CLI — напечатать в stderr и выйти с
  ненулевым кодом.

Это тот же класс бага, что описан в `config.py:14-16` («0 chunks и exit 0»):
тихий отказ хуже явной ошибки, потому что выглядит как рабочий ответ.

### Приёмка

1. Сымитировать занятый lock (открыть коллекцию в другом процессе).
2. Запустить поиск → в stderr/log видна причина «Can't lock read-write
   collection», а не пустой ответ.

## Дефект 3: MCP-ответ обрезает score до 3 знаков

**Симптом.** MCP-инструмент `rag_search` печатает `score=0.016` для всех
результатов, хотя реальные значения `0.0159-0.0164`. Создаётся иллюзия
плоских score и сломанного ранжирования.

**Корень.** `mcp_server.py:58`:

```python
f'--- Result {i} (score={r["score"]:.3f}) ---\n'
```

Формат `:.3f` округляет 4-значные score до 3 знаков. Спред 0.0005 между
top-1 и top-5 исчезает.

**Следствие в ARLINE.** Эта косметика привела к ложному диагнозу «сломанное
ранжирование», на котором была построена теория про `RrfReRanker(rank_constant=60)`,
записанная в архитектурный документ и потратившая несколько часов
расследования. Не косметика — диагностика.

### Решение

```python
# mcp_server.py:58 — заменить
f'--- Result {i} (score={r["score"]:.4f}) ---\n'
```

4 знака сохраняют дифференциацию в нашем диапазоне. CLI (`cli.py:67`) уже
печатает `:.4f` — привести MCP в соответствие.

### Приёмка

1. `rag_search "ферма воркеров" --collection docs` через MCP возвращает
   разные score (минимум 2 различных значения в top-5).

## Порядок исполнения

| # | Дефект | Сложность | Риск |
|---|--------|-----------|------|
| 1 | read-only lock | средняя (трогает `_open_or_create_collection`, надо проверить, что read-only коллекция отвечает на `query()`) | средний — если zvec в read-only не поддерживает какой-то тип запроса, поиск сломается |
| 2 | тихий отказ | низкая (логирование + текст ошибки) | низкий |
| 3 | обрезка score | тривиальная (1 символ) | нулевой |

Рекомендация: начать с дефекта 3 (тривиально, снимает ложные диагнозы в
будущем), затем 2 (даёт видимость причин на следующих этапах), затем 1
(основная работа, с тестом read-only query).

В ходе расследования высказывалась гипотеза, что `RrfReRanker(rank_constant=60)`

не трогать.
