## Результат: 8 passed, 2 failed

### Fail 1: `test_standard_cyrillic_behavior_matches_platform`

```
AssertionError: standard unexpectedly matches Cyrillic on Windows: {'ru_neg'}.
If zvec changed, the text_fts workaround may no longer be needed on this platform.
```

**Подтверждено предположение из pyproject.toml**: в 0.6.0 токенайзер
`standard` **находит кириллицу на Windows** (раньше выбрасывал целиком).
Тест был написан ровно для этого момента — упал так, как должен был.

**Влияние**: обход через поле `text_fts` с `whitespace` токенайзером
становится **избыточным** на Win/Linux. На macOS он и так был избыточным.

### Fail 2: `test_identifier_split_by_standard`

```
_insert(coll, 'id1', 'def build(): obj_k3_gab3(obj)', '')
assert _hit_ids(coll, 'text', 'gab3') == {'id1'}   # → set()
```

В 0.6.0 `standard` **перестал резать `obj_k3_gab3` по подчёркиванию** —
поиск по части `gab3` больше не находит документ. Это обратная сторона
того же юникод-улучшения: идентификаторы воспринимаются как единый токен.

**Влияние**: **регрессия для code search**. Поиск куска имени функции
(типичный сценарий отладки: «помню, было что-то про gab3») перестаёт
работать.

```
  двери            → []

```

## Главная тревожная находка: `whitespace` токенайзер сломан?

В zvec 0.6.0 `text_fts` (whitespace) **возвращает `[]` для всех запросов**.
Это не просто «обход стал избыточным» — **обход перестал работать вообще**.
При этом `standard` кириллицу находит, поэтому простой апгрейд **не
полностью ломает кириллицу**, но:

- существующий обход (`text_fts`) нужно **убрать** (он мёртвый код);
- альтернативного механизма поиска кириллицы, который работал бы через
  `standard`'s newfound ability, достаточно для **точных совпадений**
  (`коробка` → найдено), но непроверено для более сложных случаев
  (склонения, составные слова).

## Влияние на существующий код

- `fts_normalizer.py` — `make_fts_unicode` (питоновская нормализация для
  `text_fts`) становится **мёртвым кодом** при 0.6.0: поле `text_fts`
  всё равно ничего не находит. Можно удалить поле целиком из schema и
  упростить indexer/searcher.
- `test_zvec_integration.py` — канарейки нужно **переписать**: убрать
  платформозависимое ожидание, добавить новые (identifier behavior,
  whitespace sanity).
- `zvec_indexer.py:_create_collection` — упростится (одно FTS-поле вместо
  двух).
- `zvec_searcher.py` — упростится (не нужно дублировать запросы в два
  поля).
- **Все коллекции на диске — переиндексация обязательна** (формат, скорее
  всего, несовместим; это нужно отдельно проверить).

## Влияние на функциональность: +/- сводка

**Что улучшится:**
- Кириллица через `text` работает нативно на Win/Linux — больше не нужно
  дублирующее поле, нормализатор и две трассы поиска.
- На macOS поведение выравнивается (раньше стандартное поведение + обход
  сосуществовали, теперь только одно).

**Что ухудшится:**
- Поиск частей идентификаторов (`gab3` для `obj_k3_gab3`) — **теряем**.
  Это типичный code-search сценарий, и потеря ощутима.
- Существующие индексы нужно переиндексировать (60 МБ ARLINE).

## Оценка усилий

Минимально, чтобы «оно работало на 0.6.0 хотя бы как на 0.5.1»:
- Убрать `text_fts` поле + всё связанное (`fts_normalizer`, дублирование
  в searcher) — 2-3 часа.
- Обновить канареечные тесты под новое поведение — 1-2 часа.
- Подумать, как вернуть identifier-substring search (возможно, pre-split
  идентификаторов в `text` при индексации, или принять потерю) —
  неизвестно, от пары часов до отдельной задачи.
- Полная переиндексация + проверка на реальном корпусе ARLINE — ~1 час.

**Суммарно: рабочий день, плюс неопределённость с идентификаторами.**
